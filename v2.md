1. El Caos del Contenido No Estructurado (RAG & Grounding)
   El Problema: Las IAs que usan RAG (Generación Aumentada por Recuperación) tienen que "limpiar" el HTML de tu web (quitar menús, anuncios, banners). Esto es ineficiente, consume ancho de banda y a menudo hace que la IA malinterprete el contenido al mezclar el texto del artículo con el texto de un anuncio lateral.

Qué queremos lograr: Ofrecer una "vía rápida" de datos limpios. Que el plugin entregue una versión en JSON o Markdown purificado directamente a la IA, asegurando que lo que la IA "aprende" o "cita" sea exactamente lo que el autor escribió, sin ruido.

Qué NO queremos hacer: Crear una API pública abierta a todo el mundo. Solo debe ser accesible para agentes que cumplan con ciertas reglas de identificación o pago.

2. El "Embudo" del Licenciamiento Manual
   El Problema: Actualmente, si una empresa de IA quiere usar tu contenido legalmente, tiene que contactarte, negociar y firmar un contrato. Esto es imposible de escalar para miles de pequeños agentes y aplicaciones de IA que surgen cada día.

Qué queremos lograr: Automatizar la "negociación". Que el plugin exponga términos de licencia legibles por máquina (Machine-Readable Licenses). La IA "lee" los términos, los acepta técnicamente y accede al contenido bajo esas reglas de forma instantánea.

Qué NO queremos hacer: Convertirnos en una firma legal o procesar contratos PDF. El objetivo es una validación técnica de términos, no una asesoría jurídica.

3. La Indiferencia ante el Tipo de Media (Imágenes y Vídeos)
   El Problema: Los scrapers no distinguen entre un párrafo de opinión y una fotografía artística protegida por derechos de autor o un vídeo incrustado de un tercero. Se lo llevan todo por igual.

Qué queremos lograr: Granularidad total. El dueño de la web debe poder decir: "La IA puede leer mi texto para citarme, pero tiene prohibido tocar mis imágenes para entrenar generadores de arte (como Midjourney o DALL-E)".

Qué NO queremos hacer: Romper el SEO de imágenes de Google. No queremos que las fotos desaparezcan de las búsquedas humanas, solo que sean invisibles o "ilegibles" para los bots de entrenamiento.

4. La Invisibilidad ante Agentes Autónomos (Descubrimiento MCP)
   El Problema: Agentes avanzados (como Claude o asistentes autónomos) no solo navegan webs, buscan "herramientas" y "contextos". Si tu web es solo HTML, eres invisible para la nueva ola de agentes que usan el Model Context Protocol (MCP).

Qué queremos lograr: Que tu WordPress sea "descubrible" como una fuente de autoridad. Implementar un punto de conexión (Endpoint) que diga a los agentes: "Aquí hay datos verificados sobre [Tu Nicho], puedes consultarme de forma estructurada".

Qué NO queremos hacer: Permitir que los agentes controlen tu WordPress. El acceso es de solo lectura y bajo demanda; el agente nunca tiene control sobre el sitio.

5. La Falta de un Peaje Digital (Monetización Micro)
   El Problema: El contenido tiene un valor, pero no hay una forma de cobrar 0,01€ por cada consulta que una IA hace a tu base de conocimientos. O es gratis, o es un muro de pago (Paywall) que también bloquea a los humanos.

Qué queremos lograr: Un "peaje" inteligente. Si un bot comercial quiere acceder a la versión limpia y estructurada de tu contenido, debe presentar una prueba de pago o una API Key válida.

Qué NO queremos hacer: Crear una plataforma de pagos propia. Debemos apoyarnos en estándares existentes (Stripe, Lightning Network, etc.) para evitar riesgos financieros y de seguridad.

6. La Defensa Pasiva es Insuficiente (Necesidad de "Poisoning")
   El Problema: Bloquear a veces no es suficiente; las empresas de IA a veces usan datos históricos o copias de caché. Necesitamos una forma de "marcar" o "dañar" el valor de los datos si son robados sistemáticamente.

Qué queremos lograr: Envenenamiento estratégico. Si un bot detectado como "hostil" intenta extraer datos, el plugin le sirve contenido con marcas de agua invisibles o alteraciones sutiles que arruinan la precisión de su entrenamiento, pero que son imperceptibles para un humano.

Qué NO queremos hacer: Engañar a los usuarios humanos o a los buscadores legítimos. El "veneno" solo se sirve cuando hay una certeza del 100% de que el visitante es un bot extractor no autorizado.
